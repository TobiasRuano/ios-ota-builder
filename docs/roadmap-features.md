## Install experience

### F01 — Rich install page

| | |
|---|---|
| **Status** | Planned |
| **Priority** | P1 |
| **Effort** | M |
| **Phase** | 3 |

**Problem**  
The install page shows only the app name and an Install button. When multiple builds exist for the same day, testers cannot tell which version they are about to install.

**Proposed solution**  
Extend `install.html` (generated by `generate_manifest.py`) to display:

- Marketing version + build number (`CFBundleShortVersionString` / `CFBundleVersion`)
- Git branch and short commit hash
- Build timestamp (UTC)
- Configuration badge: Debug or Release
- Optional release notes block (see F05)

Reuse shared styles from `tools/ui_theme.py` so install pages match the dashboard.

**Files likely touched**

- `tools/generate_manifest.py` — accept new CLI args or read `summary.json`
- `tools/ui_theme.py` — install-card layout helpers if needed
- `agent_build_ota.sh` / `scripts/lib/common.sh` — pass metadata into manifest generator
- `docs/SETUP.md` — no config change required

**Dependencies**  
F05 (release notes) is optional; F04 (icon) can be added independently.

**Acceptance criteria**

- [ ] Install page shows version, build number, branch, commit, date, and configuration
- [ ] Page uses `ui_theme.py` styles (not legacy inline CSS)
- [ ] All metadata matches `summary.json` for that build
- [ ] Page remains usable on iPhone Safari (narrow layout)
- [ ] `itms-services://` install link still works unchanged

**Notes**  
Some older builds on disk may have legacy `install.html` without the new theme. Regenerating is optional; new builds always use the updated template.

### F02 — QR code on install page

| | |
|---|---|
| **Status** | Planned |
| **Priority** | P0 |
| **Effort** | S |
| **Phase** | 1 |

**Problem**  
Install URLs include a long `?token=...` query string. Copying from Mac to iPhone is friction-heavy.

**Proposed solution**  
Render a QR code on `install.html` encoding the full authenticated install page URL (or the `itms-services` manifest URL). Use a zero-dependency approach:

- **Preferred:** inline SVG generated by a small Python helper (no CDN, works offline on the build Mac)
- **Alternative:** `qrcode` pip package if already acceptable as a dev dependency

Show QR only on viewports `min-width: 600px` (desktop/tablet) with helper text: “Scan with iPhone camera.”

**Files likely touched**

- `tools/generate_manifest.py`
- New helper: `tools/qr_svg.py` (optional)

**Dependencies**  
None.

**Acceptance criteria**

- [ ] QR appears on install page when opened on Mac/desktop browser
- [ ] Scanning QR on iPhone opens the install flow in Safari
- [ ] Token in URL is preserved correctly (no double-encoding)
- [ ] No external network request required to render the page

**Notes**  
Do not QR-encode raw IPA URLs for install — use the `install.html` or manifest URL so Safari handles OTA correctly.

### F03 — `/latest/<project-id>` redirect

| | |
|---|---|
| **Status** | Planned |
| **Priority** | P0 |
| **Effort** | S |
| **Phase** | 1 |

**Problem**  
Every build gets a unique folder name (`2026-06-25_2310_develop-debug`). Bookmarks and agent responses must be updated after each build.

**Proposed solution**  
Add a dynamic route in `static_server.py`:

```
GET /latest/<project-id>  → 302 → /<project-id>/<newest-build>/install.html?token=...
```

“Newest” = most recent directory under `OTA-Builds/<project-id>/` that contains `app.ipa` and `summary.json` with `status: success`. Return 404 if no successful build exists.

Also expose `latest_install_url` in `print_result_json` stdout and `summary.json` optionally.

**Files likely touched**

- `server/static_server.py` — new route in `do_GET`
- `tools/ota_index.py` — helper `find_latest_build(project_id)` (reusable)
- `scripts/print_install_url.sh` — optional `--latest` flag
- `scripts/lib/common.sh` — include `latest_install_url` in JSON output
- `docs/AGENT_INSTRUCTIONS.md` — document stable URL

**Dependencies**  
None.

**Acceptance criteria**

- [ ] `/latest/my-app` redirects to the newest successful build’s install page
- [ ] Token is appended to redirect `Location` header
- [ ] Unknown project-id returns 404
- [ ] Project with no builds returns 404 with clear message
- [ ] Auth still required (401 without token)

**Notes**  
Follow the same pattern as existing dynamic routes (`/`, `/builds.json`).

### F04 — App icon from archive

| | |
|---|---|
| **Status** | Planned |
| **Priority** | P1 |
| **Effort** | M |
| **Phase** | 3 |

**Problem**  
Dashboard and install pages are text-only. Multiple apps look identical in the build list.

**Proposed solution**  
After archive, extract the largest app icon from `app.xcarchive`:

1. Locate `Products/Applications/*.app/AppIcon*.png` or parse `Assets.car` (simpler path: read `Info.plist` + known icon paths)
2. Copy a 120×120 (or 180×180) PNG to `<build-dir>/icon.png`
3. Reference in `install.html` and dashboard via `<img src=".../icon.png">`

Store `icon_url` in `summary.json`.

**Files likely touched**

- New: `tools/extract_app_icon.py`
- `agent_build_ota.sh` — call after archive
- `tools/generate_manifest.py` — embed icon in install page
- `tools/ota_index.py` — show icon in project header or build row
- `server/static_server.py` — `.png` MIME type (likely already default)

**Dependencies**  
None.

**Acceptance criteria**

- [ ] `icon.png` exists in build output after successful archive
- [ ] Install page shows app icon above title
- [ ] Dashboard shows icon next to project name or latest build
- [ ] Graceful fallback if icon extraction fails (build still succeeds)

**Notes**  
Widget extensions and asset catalogs vary; start with Apps that use standard `AppIcon.appiconset` in the main target.

## Pipeline

### F12 — Auto-increment build number

| | |
|---|---|
| **Status** | Planned |
| **Priority** | P0 |
| **Effort** | M |
| **Phase** | 1 |

**Problem**  
Reinstalling an OTA build with the same `CFBundleVersion` can confuse iOS or skip updates. Manually bumping build numbers before each OTA build is tedious.

**Proposed solution**  
Merge branch `feature/auto-increment-build-number` and integrate `scripts/resolve_build_number.sh` into the pipeline **before** archive:

1. Read current `CFBundleVersion` from Xcode project
2. Compare with last successful OTA build and persisted counter (with `flock`)
3. Bump to `max(project, ota, stored) + 1`
4. Write back to project (or use `agvtool` / `PlistBuddy` as implemented on branch)
5. Fail on non-integer `CFBundleVersion`

**Files likely touched**

- `scripts/resolve_build_number.sh` (exists on feature branch)
- `agent_build_ota.sh` — call before `build_archive.sh`
- `docs/AGENT_INSTRUCTIONS.md` — note that pipeline may modify build number only
- `.gitignore` — counter state file if used

**Dependencies**  
None. **First actionable step:** merge or cherry-pick `feature/auto-increment-build-number`.

**Acceptance criteria**

- [ ] Each successful build has a strictly increasing integer `CFBundleVersion`
- [ ] Concurrent builds serialize counter updates (`flock`)
- [ ] Counter revalidates against OTA `summary.json` history
- [ ] Non-integer bundle version fails with clear error
- [ ] Documented in SETUP.md as intentional pipeline side effect

**Notes**  
This is the only planned feature that touches project source (build number only), contradicting the general “pipeline does not modify source” rule — document the exception.

### F14 — Per-project build lock

| | |
|---|---|
| **Status** | Planned |
| **Priority** | P2 |
| **Effort** | S |
| **Phase** | 4 |

**Problem**  
Two simultaneous builds for the same app can corrupt DerivedData or race on export.

**Proposed solution**  
Acquire `flock` on `$OTA_BUILDS_DIR/.lock-<project-id>` at start of `agent_build_ota.sh`; release on exit. Second invocation waits or fails immediately with `OTA_BUILD_LOCK=wait|fail` (default: fail fast with message).

**Files likely touched**

- `agent_build_ota.sh`
- `scripts/lib/common.sh` — `acquire_build_lock` / `release_build_lock`
- `config/local.env.example`

**Dependencies**  
Shares locking pattern with F12 counter file.

**Acceptance criteria**

- [ ] Parallel builds for **different** project-ids proceed normally
- [ ] Second build for **same** project-id fails (or waits) with clear message
- [ ] Lock released on success and failure (trap handler)

**Notes**

## Shared implementation notes

### Dynamic vs static pages

The OTA server already serves a **dynamic** dashboard (`/` → `ota_index.render_index`). New routes (F03, F18, F24, F26) should follow the same pattern in `server/static_server.py` rather than generating static HTML on each build where freshness matters.

### `summary.json` schema evolution

Several features extend `summary.json`. New fields should be optional so older builds still render. Document the canonical schema in `docs/AGENT_INSTRUCTIONS.md` when implementing.

### Auth and tokens

Features that expose URLs (F02 QR, F06 copy, F15 notifications) must not leak tokens into logs or public notification centers. Webhooks should use a dedicated secret header, not the OTA access token.

### Testing checklist (per feature)

1. Build succeeds end-to-end with `agent_build_ota.sh`
2. Dashboard renders with token auth
3. Install works on physical iPhone via Safari
4. `audit-public-safe.sh` passes before any public push
